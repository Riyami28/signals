# Watchlist & Account Ingestion

How company accounts get into the system — from CSV files, Wikidata, and the web UI.

## How Accounts Get Into the DB

Every pipeline command (`run-daily`, `ingest`, etc.) calls `_bootstrap()` in `src/main.py` before doing any work. `_bootstrap()`:

1. Calls `db.init_db(conn)` — creates tables if not present.
2. Calls `db.seed_accounts(conn, seed_accounts_path)` — reads `config/seed_accounts.csv`.
3. Calls `db.seed_accounts(conn, watchlist_accounts_path)` — reads `config/watchlist_accounts.csv`.

`seed_accounts()` is a pure upsert: it skips any domain already in the `accounts` table. Safe to run repeatedly.

```
config/seed_accounts.csv      →  _bootstrap()  →  accounts table (signals schema)
config/watchlist_accounts.csv →  _bootstrap()  →  accounts table (signals schema)
```

**`migrate` does NOT seed accounts.** `migrate` is schema-only. Account seeding only happens when a pipeline command runs.

## The Watchlist Files

### `config/seed_accounts.csv`
19 hand-picked companies used as ICP reference accounts. Always committed to git.

Columns: `company_name, domain, source_type`

### `config/watchlist_accounts.csv`
Target companies sourced from Wikidata. The **1,000-row starter set** is committed to git. The full 143K file is too large for git — each developer generates it locally (see below).

Columns: `company_name, domain, source_type, country, region_group, industry_label, website_url, wikidata_id, sitelinks, revenue_usd, employees, ranking_score, data_source, last_refreshed_on`

> Do NOT edit this file by hand. It has 143K rows in production and is managed by the `build-cpg-watchlist` command.

## Regenerating the Full Watchlist from Wikidata

The watchlist is built by querying the [Wikidata SPARQL endpoint](https://query.wikidata.org/) for CPG (Consumer Packaged Goods) companies by country.

```bash
# Rebuild watchlist_accounts.csv from Wikidata (requires internet)
./signals build-cpg-watchlist --limit 143000

# Optional: skip merging into account_source_handles.csv
./signals build-cpg-watchlist --limit 143000 --no-merge-handles
```

**Output:**
```
requested_limit=143000 raw_rows=... deduped_rows=... selected_rows=...
handles_inserted=... failed_country_count=0 watchlist_path=config/watchlist_accounts.csv
```

**What it does:**
1. Queries Wikidata SPARQL for CPG companies, grouped by country.
2. Deduplicates by domain (normalised).
3. Ranks by `ranking_score` (composite of revenue, employees, sitelinks).
4. Writes `config/watchlist_accounts.csv`.
5. Merges generated domains into `config/account_source_handles.csv` (unless `--no-merge-handles`).

**Time:** ~5–20 minutes for 143K rows depending on Wikidata response times.

**Failure recovery:**
- `failed_country_count > 0` — Wikidata timed out for some countries. Re-run; it will fill gaps.
- `raw_rows=0` — Wikidata is down or rate-limiting. Try again in a few minutes.
- Partial file — the command writes atomically; a failed run does not corrupt the existing file.

After regenerating, accounts load on the **next pipeline run** via `_bootstrap()`.

## Developer Workflow: Fresh Machine

```bash
make setup                             # schema only — no accounts yet
./signals start --date $(date +%Y-%m-%d)  # first run seeds 1K starter watchlist

# optional: upgrade to full 143K watchlist
./signals build-cpg-watchlist --limit 143000
./signals start                        # next run seeds the extra accounts
```

## Checking How Many Accounts Are in the DB

```bash
docker exec signals-postgres psql -U signals -d signals \
  -c "SELECT COUNT(*) FROM signals.accounts;"

# breakdown by source_type
docker exec signals-postgres psql -U signals -d signals \
  -c "SELECT source_type, COUNT(*) FROM signals.accounts GROUP BY source_type;"
```

## UI Ingestion Path

> **Status:** CSV upload (#37 ✓ done) and pipeline trigger (#38 in progress). The Wikidata watchlist refresh from the browser is not yet implemented — tracked as part of epic #15.

### Current UI flow (once #37 + #38 are complete)

```
Browser → POST /api/v1/upload/csv  →  batch stored (upload_batches + batch_companies)
        → POST /api/v1/pipeline/run  →  ingest + score runs for that batch
        → GET  /api/accounts          →  view scored results
```

1. **Upload CSV** — go to the web UI, click Upload, select a CSV with at least `company_name` and `domain` columns. Column names are auto-detected (Claude AI fallback for unknown headers). Max 10MB / 10K rows.
2. **Trigger pipeline** — after upload, click "Run Pipeline" for the batch. Supports selecting stages (`ingest`, `score`, `enrich`, `export`).
3. **View results** — accounts appear in the dashboard with scores and tier assignments.

### Planned: Wikidata watchlist refresh from UI (epic #15)

To allow non-CLI users to refresh the full account list, issue #38 should expose:
```json
POST /api/v1/pipeline/run
{ "source": "watchlist", "limit": 143000 }
```
This would call `build-cpg-watchlist` then trigger seeding, all from the browser. Not yet built.

## Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| `make migrate` reports 10 accounts imported | Old bug — fixed in d7af491. `migrate` no longer seeds. | Update to latest main. |
| Accounts table empty after `make setup` | Expected — seeding is deferred to first `./signals start`. | Run `./signals start`. |
| `build-cpg-watchlist` returns `raw_rows=0` | Wikidata SPARQL is down or rate-limiting. | Wait a few minutes, retry. |
| `failed_country_count > 0` | Some countries timed out in Wikidata query. | Re-run; successful countries are skipped. |
| `seed_accounts` inserts 0 rows | All accounts already in DB. | Expected on repeat runs — check `SELECT COUNT(*) FROM signals.accounts`. |
| Watchlist CSV has 1K rows, not 143K | Starter set is committed; full list must be generated. | Run `./signals build-cpg-watchlist --limit 143000`. |
